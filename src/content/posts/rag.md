## RAG 是什么？

RAG = Retrieval-Augmented Generation（检索增强生成）

一句话：让大模型在回答问题之前，先去查资料，再基于查到的内容来回答。

```
用户提问 → 检索外部知识 → 将检索结果 + 问题一起喂给大模型 → 生成回答
```

## 大模型的两个结构性缺陷

| 缺陷 | 本质 | 表现 |
|------|------|------|
| **知识固化** | 参数即知识，训练完就锁死在 cutoff 之前 | 不知道最新发生的事，更新需重新训练 |
| **无法溯源** | 生成过程是黑盒概率采样 | 答案从哪来不知道，所以容易 hallucinate，也无法验证 |

## RAG 恰好对症下药

```
┌─────────────────┐          ┌─────────────────┐
│    知识固化      │    →    │   外挂知识库     │
│   (参数锁死)    │          │  (动态更新文档)  │
└─────────────────┘          └─────────────────┘

┌─────────────────┐          ┌─────────────────┐
│    无法溯源      │    →    │   检索+引用      │
│   (黑盒生成)    │          │  (答案有据可查)  │
└─────────────────┘          └─────────────────┘
```

## RAG 的工作流程

```
┌─────────────┐      ┌─────────────┐      ┌─────────────┐
│   用户提问   │  →   │  向量检索    │  →   │  大模型生成  │
│"公司Q3财报   │      │从知识库找    │      │基于检索结果  │
│ 营收多少？"  │      │相关文档片段  │      │组织语言回答  │
└─────────────┘      └─────────────┘      └─────────────┘
                            ↑
          └──── 检索到的内容作为上下文 ────┘
```

## 关键组件

| 组件 | 作用 |
|------|------|
| **知识库** | 存放文档（PDF、网页、数据库等），可动态更新 |
| **Embedding 模型** | 把文字变成向量，实现语义检索 |
| **向量数据库** | 存储和检索向量（Milvus、Pinecone、Chroma、PGVector） |
| **大模型** | 基于检索到的内容生成带引用的回答 |
| **重排序（Reranker）** | 对检索结果精排，提升相关性 |

## 典型应用场景

| 场景 | 例子 |
|------|------|
| 企业知识库问答 | 员工问"报销流程是什么"，从内部文档找答案并标注条款 |
| 客服机器人 | 基于产品手册回答，不胡说，且能告诉用户答案来自哪页 |
| 法律/医疗咨询 | 引用真实法条/病例，降低风险，可核查 |
| 个人知识管理 | 笔记、收藏、论文，变成可对话、带引用的知识库 |
| 代码助手 | 基于项目代码库回答，指出具体文件和行号 |

## 进阶形态

| 技术 | 解决什么问题 |
|------|------|
| **Query 改写** | 用户问得模糊，改写成更适合检索的形式 |
| **多路检索** | 向量检索 + 关键词检索 + 知识图谱，互补召回 |
| **Rerank** | 粗排结果不精准，用更重的模型二次排序 |
| **Self-RAG** | 模型自己判断「需不需要检索」「检索够不够用」 |
| **GraphRAG** | 结合知识图谱，处理复杂多跳关系（如"A 的 B 的 C 是什么"） |

## 一句话总结

**RAG 不是让模型变聪明，是让模型开卷考试，且必须标注出处。**
